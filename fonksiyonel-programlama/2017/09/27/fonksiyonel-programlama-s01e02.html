S01E02:

Previously on FP

FP, matematikteki fonksiyon tanımındaki fonksiyonları kullanılarak yapılan programlamadır.

Fonksiyon olması için;

  1. A’nın her elamanı B’ye gidecek.
  2. A kümesinde açıkta eleman kalmayacak.
  3. A’nın herhangi bir elamanı B’ye iki defa gitmeyecek.

Not: FP literatüründe buna pure functions deniyor. Bir yerlerde görürseniz şaşırmayın diye belirteyim istedim.

Normal bir program

Herkes javascript bilir. Ama anlatmak istediğim şeyi anlatabilmek için biraz typescript'ten yararlanacağım. Çünkü javascript'in süper esnekliği burada bana yardımcı olmayacak.

TypeScript is a free and open-source programming language developed and maintained by Microsoft. It is a strict syntactical superset of JavaScript, and adds optional static typing to the language.

Neyse ki siz okuyucularım için docker üzerinden de çalıştırılabilir hale getirdim. Sizin için daha mı kolay daha mı zor tartışılır. Çok da umrumda değil; benim için kolay olacağı kesin. Ama bir kere docker kurduğunuz zaman tüm örnekleri onun üzerinden çalıştırabilirsiniz.

Kodlara ayrıca github üzerinden de ulaşabilirsiniz. Docker Hub'dan indirmek için aşağıdaki komutu uygulayın:

1 docker pull dhalsim/fp-exercises-s01e02

Ardından docker container'ın interactive shell'ini şu komutla çalıştırabilirsiniz:

1 docker run -it dhalsim/fp-exercises-s01e02

Egzersizler

Gönlüm el vermedi ve programın yapısını size Bilal'e çizer gibi - çizim programı öğreniyorum en azından diye kendimi avuta avuta - çizdim:

Hadi source kodu incelemeye başlayalım.

1 ~ $ cat exercise01.ts

 1 export interface User {
 2     id: number;
 3     name: string
 4 }
 5 
 6 export let users: [User] = [
 7     { id: 1, name: "halide" },
 8     { id: 2, name: "edip" }
 9 ];
10 
11 export let getUserById = (id: number): User => users.find(x => x.id === id);

Neler var neler? User diye bir tip tanımladım. number tipinde id ve string tipinde name özellikleri mevcut.

Ayrıca users isminde bir User array tanımladım.

Son olarak getUserById isimli bir fonksiyon tanımlıyoruz. Parametre olarak number tipinde bir değişken alıp, es6 target'inde desteklenen Array.prototype.find() fonksiyonunu kullanıyoruz.

The find() method returns the value of the first element in the array that satisfies the provided testing function. Otherwise undefined is returned.

Burada dikkat edilecek husus, find sonucunda istenen değer bulunamazsa undefined dönüyor olması. Bu ne demek? Bu bizim yukarıda yaptığımız matematiksel fonksiyon tanımına uymuyor demek. Yani uygun bir parametre - number tipi - gönderdiğimizde bile her zaman bize User tipinde bir sonuç dönmüyor.

Bu durumu şöyle açmak lazım: fonksiyonumuz aslında tam olarak dürüst değil. Yani fonksiyonun imzasına uygun hareket etmiyor. Bu da bu fonksiyonun kullanımında özellikle composition'da problemlere yol açacak demektir.

Daha önce de söylemiştik: FP'nin temeli composition'dır.

Şimdi programızı test edelim. typescript compiler çalıştırıyoruz. Merak ediyorsanız tsconfig.json dosyasından konfigürasyona da bakabilirsiniz.

1 ~ $ tsc

.ts dosyalarımız derlendi ve .js dosyaları oluştu. Son olarak da exercise02.js dosyasını görelim:

1 ~ $ cat exercise02.ts

1 import { User, users, getUserById } from "./exercise01";
2 
3 let getUserName = (user: User): string => user.name
4 let getGreatingOfName = (name: string): string => "hello " + name;
5 
6 console.log(getGreatingOfName(getUserName(getUserById(1))))
7 console.log(getGreatingOfName(getUserName(getUserById(3))))

Neler yapmışız?

getUserName isminde bir fonksiyon tanımladık. Parametre olarak aldığı User objesinin name özelliğini döndürüyor.

Çalıştırma ve çıktısı:

1 ~ $ node exercise02.js

Neden?

Hata almamız gayet normal çünkü users array'inde 1 id'li User tanımlı ama 3 id'li User tanımlı değil. undefined dönen sonucu getUserName fonksiyonuna geçiriyoruz ve sonucunda undefined bir değerin name özelliğine ulaşmaya çalışırken TypeError alıyoruz.

Şimdi typescript'in ayarını düzenleyip bir daha deneyelim:

1 tsc --strictNullChecks

1 exercise01.ts(11,48): error TS2322: Type 'User | undefined' is not assignable to type 'User'.
2   Type 'undefined' is not assignable to type 'User'.

Typescript compiler eğer doğru konfigürasyonda çalıştırılırsa bize zaten null kontrolünü yapıp problemli yerlerde bunun uyarısını yapıyor.

Null References: The Billion Dollar Mistake:

Tony Hoare introduced Null references in ALGOL W back in 1965 "simply because it was so easy to implement", says Mr. Hoare. He talks about that decision considering it "my billion-dollar mistake".

Çoğu FP dillerinde null referansının olmaması veya teşvik edilmemesinin geçerli bir sebebi var. Ancak bir ihtiyaçtan kaynaklandığı da doğru.

FP'de nasıl çözülür?

FP camiasında the M word denerek adı anılmayan bir terim var. Matematikteki kategori teorisinden alınan bu terim monad'dır. Şimdi anlatacağım yapı: Option da bir monad'dır. Ama hiçbir tanım yapmadan direkt örnek üzerinden gireceğim.

Belki burada bir Monad vardır: Option

Bir değerinizin olduğunu düşünün. Integer, string veya custom bir tip olsun farketmez. Alabileceği iki farklı durum vardır:

  • Kendi değeri: 5, "Ali" veya User objesi. Farklı dillerde Some, Just gibi ifade edildiği olmuştur.
  • Veya değerinin olmaması. Farklı dillerde farklı şekilde ifade edilir: None, Nothing gibi.

Şimdilik işin felsefesine teorisine çok karışmadan örnek üzerinden bakalım. Normal koşullarda exercise02.js'de aldığımız hatayı düzeltmek için yapabileceğimiz şeyler kısıtlıdır. Bunlardan en pratiği null kontrolü yapmaktır. exercise03.ts dosyasında bunu yaptım, inceleyelim:

Bilinen çözüm:

1 ~ $ cat exercise03.ts

 1 import { User, users, getUserById } from "./exercise01";
 2 
 3 let getUserName = (user: User): string => {
 4   if(user) {
 5     return user.name
 6   } else {
 7     return "";
 8   }
 9 };
10 
11 let getGreatingOfName = (name: string): string => "hello " + name; 
12 
13 console.log(getGreatingOfName(getUserName(getUserById(1))))
14 console.log(getGreatingOfName(getUserName(getUserById(3))))

Derleyip çalıştıralım:

1 tsc
2 node exercise03.js

1 $ node exercise03.js 
2 hello halide
3 hello

Haydaa!

getGreatingOfName'de de şimdi boş string kontrolü yapmam gerekiyor. Onu yapsam bu sefer console.log boşuna çağırılacak. Bir if de oraya koymalı.

İşte arkadaşlar null kullanımının zararları. Basit diye düşündüğümüz tek satırlık fonksiyonlar bir anda şişmeye başlıyor. Neyse ki FP'de buna şık bir çözüm bulabiliyoruz:

FP ile çözüm:

Option veya diğer dillerde geçen haliyle Maybe. Ben bunu Schrödinger'in Kedisi'ne benzetiyorum. İçinde tuttuğu değeri bizden saklıyor ve biz bunun kesin sonucunu fonksiyonu çağırıp çıktıyı basana kadar bilemiyoruz.

Bizi bu bilinmez durumun farkında olmamızı sağlıyor. Yani null kontrolü unutmuşuz hadi düzeltip bir daha kod atıyoruz gibi bir durumla karşılaşmıyorsunuz çünkü - en azından strongly-typed dillerde - o kontrolü yapmadan kullanamıyorsunuz.

Haydi exercise04.ts

1 ~ $ cat exercise04.ts

 1 import { User, users } from "./exercise01";
 2 import { Some, None, Option, Maybe } from "./option";
 3 
 4 let getUserById = (id: number): Option<User> => Maybe(users.find(x => x.id === id));
 5 
 6 let getUserName = (user: User): string => user.name;
 7 
 8 let getGreatingOfName = (name: string): string => "hello " + name;
 9 
10 getUserById(1).map(getUserName).map(getGreatingOfName).map(console.log);
11 getUserById(3).map(getUserName).map(getGreatingOfName).map(console.log);

getUserById fonksiyonunu exercise01.ts'ten almayıp farklı bir versiyonunu kendimiz yazdık. Dönüş tipi artık User değil Option<User> oldu.

Diğer değişen kısımlar da aşağıdaki fonksiyon çağrımları oldu. Farkettiğiniz üzere map diye bir fonksiyon kullanıyoruz ve bu Option monadından geliyor. Farkettiniz mi bilmiyorum ama bu map aynı zamanda javascript Array üzerinde de tanımlıdır ve size bir sır vereyim: Array de bir Monad'dır.

Programımızın yeni halini çalıştıralım:

1 ~ $ node exercise04.js 
2 hello halide

Gördüğünüz gibi ikinci satır hiç çalışmadı. Nedenini sıradaki bölümde öğreneceğiz.

Karşınızda Option Monad!:

Şimdi de option.ts'yi inceleyelim:

1 ~ $ cat option.ts

 1 export interface Option<A> {
 2     map<B>(fn: (a: A) => B): Option<B>;
 3 }
 4 
 5 export class Some<A> implements Option<A> {
 6     value: A
 7 
 8     constructor(value: A) {
 9         this.value = value;
10     }
11 
12     map<B>(fn: (a: A) => B): Option<B> {
13         return new Some(fn(this.value));
14     }
15 
16     static of(value){
17         return new Some(value);
18     }
19 }
20 
21 class NoneClass implements Option<any> {
22     map<B>(fn: (a: any) => B): Option<B> {
23         return None;
24 
25     }
26 }
27 
28 export const None = new NoneClass();
29 
30 export const Maybe = function OptionObject<T>(value: T): Option<T> {
31   return (value !== null && value !== undefined) ? Some.of(value) : None
32 }

Nasıl Schrödinger'in kedisini, kutuyu açmadan canlı mı ölü mü olduğunu bilemiyorsak, Option için de aynısı geçerli. Fonksiyonlarımız map üzerinden çağrılmadan değerimize ulaşamaz.

Önceden de dediğimiz gibi Option'ın iki durumu söz konusu:

  • Some: İçinde kendi değerini taşır
  • None: Değer taşımaz

Yalnız bi kere değerimizi kutunun içine koyduktan sonra artık onu normal fonksiyonlarımızda kullanmamız mümkün değil. Çünkü kutunun içinde ve tipi değişmiş oldu. Bunu aşmanın yolu da map fonksiyonuyla oluyor.

Aşağıdaki şemadan Option monad nasıl çalıştığını görebilirsiniz:

map bir A -> B (A'dan B'ye diye okunur) bir fonksiyon alıyor. Kutuyu açıp değeri alıp fonksiyona veriyor ve sonucu hesaplayıp yeni bir kutuya koyuyor.

Yeni değerimiz kutu içinde kalmaya devam ettiği için de biz fonksiyonlarımızı aynı şekilde map üzerinden kullanmaya devam edebiliyoruz. x.map(..).map(..).map(..) şeklindeki kullanım bu şekilde çalışıyor.

Monad'larda Option örneğinden konuşacak olursak fn hesaplaması Some değerleri için çalıştırırken, None da çalıştırılmaz; yeni kutuya None olarak doğrudan geçirilir. Bu ne demek? Eğer bir kez None değeri dönülürse, zincirdeki hiçbir hesaplama çalışmayacaktır.

Bölüm sonu canavarı

Pratik kazancımız: artık fonksiyonların içinde tek tek null kontrolü yapmamıza gerek kalmadı, bu özelliği kullandığımız data üzerinden sağlayabiliyoruz.

Bu bölümde çok fazla teknik ayrıntıya girip kafaları karıştırmak istemedim. Ancak şuna dikkati çekmek lazım ki OO dillerde yapamadığımız, dil üzerindeki yapılarda bile geçerli abstaction'ı FP yöntemlerinde yapabiliyoruz.

Bunu da çok basit bir şekilde fonksiyonlarla yapıyoruz. Bu fonksiyonel yöntemlerin çoğu modern OO dillerinde de destekleniyor.

Yani bu özellikleri kullanmak için illa bir FP diline geçmeniz gerekmiyor. Geçerseniz sizin yararınıza olacaktır çünkü doğru bir FP dili sizi bu özellikleri kullanmaya zorlamalıdır. Bu şekilde öğrenmeniz de hızlanır.

Bir sonraki bölümümüzde composition meselesine biraz daha eğileceğiz. Option monad'imizi biraz daha geliştireceğiz.
